# Оценочные тесты AI-lawyer

Офлайн-проверка ключевых узлов ассистента на эталонных кейсах (107 примеров). Прогон: `make_dataset.py` → `run_test.py` → `make_metrics.py` (детали — в `metrics.json` каждого модуля).

Зависимости: `pip install -r tests/requirements.txt`, для LLM-тестов — `src/.env`.

---

## router_classification

**Суть:** куда направить запрос пользователя — иск, претензия, общий вопрос или договор.

| Метрика | Значение |
|---------|----------|
| Accuracy | **96%** (24/25) |
| Macro F1 | 0.97 |
| По сложности | easy 100%, medium 100%, hard 67% (1 ошибка) |
| Ошибка | 1 запрос на иск классифицирован как претензию (двусмысленный кейс) |

## generator_qa

**Суть:** сгенерировать иск/претензию по готовому state и пройти QA.

| Метрика | Значение |
|---------|----------|
| QA пройден | **100%** (6/6) |
| Чек-лист ключевых фраз в тексте | **83%** |
| Средняя длина документа | ~3340 символов |

Замечание: в кейсе «иск ООО к ООО (арбитраж)» в текст попала только 1 из 4 ожидаемых фраз, при этом QA всё равно прошёл.

## memory_summarizer

**Суть:** сжать диалог в структурированную сводку с сохранением фактов.

| Метрика | Значение |
|---------|----------|
| Ключевые слова в сводке | **92%** |
| Все обязательные секции (ФАКТЫ, РЕШЕНИЯ, …) | **50%** (5/10) |
| Сжатие (сводка / исходный текст) | в среднем ~2.8× |

Слабое место: не во всех диалогах модель выдерживает полный шаблон секций.

## Запуск

```bash
python tests/<модуль>/make_dataset.py
python tests/<модуль>/run_test.py
python tests/<модуль>/make_metrics.py
```

`calc_validation` — без API LLM.
